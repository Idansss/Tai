# Backend Testing

Vitest is the baseline runner. Tests sit beside the code they exercise and assert behaviour, failure paths, and boundary conditions. `pnpm test` runs all workspace suites; `pnpm check` adds formatting, linting, type checking, production builds, and Prisma validation.

Later phases add database integration tests against isolated PostgreSQL, permission matrices, webhook fixtures, queue/job tests, contract tests, and API end-to-end tests. Mandatory commerce concurrency and duplicate-event scenarios remain tracked in `docs/progress/BACKEND_TODO.md`.

TMS-B1-002 adds a real HTTP integration suite against disposable PostgreSQL 17. It covers registration validation and duplicate accounts, hashed credentials/tokens, one-time verification, production cookie options, indistinguishable login/recovery failures, session authentication, object-level revocation, revoke-all, password-reset invalidation, logout idempotency, throttling, and audit evidence. Pure unit tests cover scrypt/HMAC helpers, cookie scope, email rendering, error mapping, and shared auth schemas.

TMS-B1-003 adds a second real HTTP/PostgreSQL suite for staff authentication. It verifies safe credential failures, customer/admin audience isolation, cookie scope, live roles/permissions, TOTP enrollment, encrypted secret storage, one-time challenges, replay rejection, MFA assurance, permission denial, dynamic role grants, cross-admin session boundaries, Owner-only elevation, final-Owner preservation, provisioning, audited operator MFA reset, and denial audits. Pure tests use the RFC 6238 SHA-1 vectors and cover base32, authenticated encryption, time windows, and `otpauth` URIs. The identity suite now deploys both migrations twice and verifies the new indexes and assurance constraint.

TMS-B2-001 adds six real HTTP/PostgreSQL artwork scenarios: authentication and read/write permission separation, validated draft creation and slug conflict, database/HTTP mutation bypass attempts, ordered version creation, exact publication and prior-version archival, public draft isolation, cursor pagination, cross-artwork identifier rejection, archive/republish behavior, and audit evidence. The persistence suite deploys all three migrations twice and directly verifies immutable-content/delete triggers, lifecycle constraints, the one-published-version index, and catalogue indexes.

TMS-B2-002 adds six catalogue HTTP/PostgreSQL scenarios covering separate read/write permission grants, tag CRUD and compound tag-kind filtering, collection lifecycle and draft-member isolation, timed drops, limited editions, narrative search, ordered story replacement, public draft isolation, safe validation/conflict errors, mutation audits, and deletion. The persistence suite deploys all four migrations and directly rejects invalid drop windows, numbered editions without quantities, stories with two parents, and duplicate story-block positions while checking the discovery indexes.

TMS-B2-003 adds six garment HTTP/PostgreSQL scenarios covering authentication and read/write permission separation, incomplete publication rejection, measured sizes, cross-template variants, duplicate SKUs, normalized placement geometry, placement/scale publication ordering, public draft isolation, artwork compatibility approval, placement allowlists, exact configuration validation, type filtering, safe errors, and mutation audits. The persistence suite deploys all five migrations and directly rejects invalid measurements, cross-template variant members, invalid geometry, invalid lifecycle timestamps, and cross-template compatibility placements while checking garment indexes. Migration and seed commands remain individually bounded; the suite allows a five-minute setup window so full-workspace API/database runs can share Docker without false hook expiry.
TMS-B2-004 adds pure image tests for magic/MIME/extension agreement, dimensions, metadata, checksums, transparency, low-resolution flags, derivatives, and the EICAR scanner hook; API service tests cover scan-before-storage, persistent job-before-queue ordering, queue failure state, and mockup approval; worker tests cover deterministic outputs, idempotent repository completion, and retry-visible failures. PostgreSQL integration deploys all six migrations twice and verifies asset immutability, exact-version source membership, approval constraints, and media indexes.
